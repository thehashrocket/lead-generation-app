## Problem Statement

Jason has built a volunteer match app that is live but has not yet activated consistent users. He needs to do outbound outreach to non-profits (animal shelters, volunteer organizations, programs that need volunteers) to acquire orgs as users for the app. He has done zero cold outreach to date and wants a tool he can use daily.

The original concept was a Chrome extension that drives a logged-in LinkedIn session to crawl LinkedIn for matching businesses and contacts, with an API back to a host app. Two issues with that concept surfaced during this session:

1. LinkedIn is the wrong primary source for non-profit org discovery. Public 990 data (ProPublica API summary search + IRS bulk Form 990 XML for mission text) is free, queryable, and structurally richer for non-profits than LinkedIn's profile pages.
2. LinkedIn Chrome-extension scraping in 2026 carries account-suspension risk. Q1 2026 LinkedIn fingerprinting upgrades, the March 2026 HeyReach ban (cited via search; verify before relying on as load-bearing), and the April 2026 "BrowserGate" disclosure that LinkedIn scans for 6,000+ Chrome extensions all point the same direction. (See Open Questions for source verification.)

## What Makes This Cool

Most prospecting tools (Apollo, Saleshandy, ZoomInfo) target generic B2B and use scraped LinkedIn data. For non-profits specifically, public 990 filings contain richer data than LinkedIn: mission statements, program descriptions, total revenue, executive compensation, named officers, recent grants. An LLM personalizing outreach against parsed 990 data can produce dramatically more relevant emails than any general-purpose tool, because the data is structurally better for this niche.

The 990 data + LLM combination also unlocks a "lookalike" angle: study the orgs already on the volunteer match app, find peers with similar NTEE codes and revenue profiles, pitch them with social proof.

## Constraints

- Single user (Jason) for the foreseeable future. No multi-tenant concerns.
- Must use Jason's real Gmail inbox to send (not a mass-mail service) so replies route normally.
- LinkedIn touch must be indistinguishable from normal browsing. Manual one-click capture only, no DOM crawling, no automated navigation.
- Pre-PMF for the volunteer match app, so the tool's job is to help Jason run his first 20 outreaches with structure, not to scale to 1000s.
- 3-week build budget, shippable in stages so Jason has a usable tool from end of week 1.
- Hard cap: 50 cold emails/week from Jason's real Gmail to maintain deliverability and avoid spam-flag risk.
- Gmail API: stay in OAuth "Testing" mode with self as the only test user. Tokens expire every 7 days; Jason re-consents weekly. Production verification (which can take weeks) is explicitly not in scope for v1.

## Premises

1. For non-profit org discovery, the ProPublica Nonprofit Explorer API (free, public, returns EIN, name, NTEE, state, total revenue, and links to 990 PDFs) is sufficient for the search/filter step. Full mission/programs text comes from parsing IRS bulk Form 990 XML (Part III) on demand for selected orgs.
2. A Chrome extension that crawls LinkedIn pages while logged in is too risky to build in 2026. If LinkedIn is used at all, it must be for manual one-click capture of a single page Jason is already viewing.
3. Jason has never sent a cold email, so the tool's job is "structured first 20 outreaches that teach me what works," not "scaled outreach to hundreds."
4. The success metric is not "leads found." It is "quality conversations with the right person at the right org that teach Jason whether the volunteer match app solves a real problem."

All four premises were agreed during this session.

## Approaches Considered

### Approach A: The 30-Minute Test (minimal viable)
No code. ProPublica web search + spreadsheet + Gmail. Manual outreach to 20 orgs in one hour. Validates the channel before any tool exists.

### Approach B: The 990-Powered Personal Pipeline (recommended, with A in parallel for week 1)
Small Next.js app. ProPublica API for org discovery, IRS 990 XML parsing for mission/programs text, LLM-personalized email drafts, Gmail API for send-from-inbox, Postgres for tracking. LinkedIn used only for manual one-click contact capture via a minimal Chrome extension that copies text from a profile page Jason is actively viewing.

### Approach C: Lookalike Outreach From Existing Users (creative)
Use the few orgs already on the volunteer match app as seeds. Find 30 lookalikes per seed via ProPublica (same NTEE code, similar revenue, region). LLM drafts cite the seed org as social proof.

## Recommended Approach

**Approach B as the headline build, with Approach A running in parallel during week 1.**

Approach A is mandatory because Jason has done zero cold outreach. Sending 5-20 hand-drafted emails this week teaches him what makes orgs respond before any tool is shaped. Without that signal, the tool will optimize the wrong things.

Approach B is the right shape for "a tool I use daily." It ships in stages, so a usable v1 (org search + basic draft generation) lands at end of week 1.

Approach C is deferred until the volunteer match app has at least 2-3 active orgs to seed from. It can be added as a layer on top of Approach B in a later iteration.

### Architecture Sketch (Approach B)

**Stack notes.** Next.js 16 App Router on Vercel with React Server Components, shadcn/ui, and TailwindCSS 4. Node 24 LTS via NVM (`.nvmrc` committed day 1; `"engines": {"node": ">=24.0.0"}` in `package.json`). Vercel AI Gateway for LLM access. Postgres via Vercel Marketplace (Neon) with **Drizzle ORM + drizzle-kit** for type-safe queries and schema migrations. **Biome** for linting + formatting. **T3 Env** (`@t3-oss/env-nextjs`) for typed env var validation at startup. **Vitest + Playwright** for unit/integration and E2E tests. TypeScript strict mode enforced from day 1. For a single-user tool, this is heavier than required (a SQLite file + plain Express would also work), but the chosen stack is what Jason already uses and the Vercel skills in this session are tuned for it.

**Frontend.** Three views: Search, Drafts, Sent/Tracking. Single-tenant. Auth: a single env-var-gated middleware check for the web UI (not Vercel Authentication, which complicates extension calls). Extension uses a separate API route with a long-lived token.

**Org discovery (ProPublica).** Server Action calls `https://projects.propublica.org/nonprofits/api/v2/search.json`. No auth key required, but rate limit applies; cache responses in Postgres for 24h. Filter by NTEE code, state, revenue range. Returns summary fields only (EIN, name, NTEE code, state, total revenue, link to 990 PDF), not free-text mission. This is enough for week 1.

**990 mission/programs enrichment (week 2).** On org-detail view, fetch the org's most recent Form 990 XML from the IRS bulk dataset (via ProPublica's per-org XML endpoint first; fall back to the IRS S3 mirror at `s3://irs-form-990` year-index lookup only if ProPublica returns 404). Parse Part III (Statement of Program Service Accomplishments) for mission text and program descriptions. Cache parsed text in Postgres. If XML is unavailable for an org (older or non-electronic filers), fall back to ProPublica summary only and flag the org as "low-context" in the UI.

**LLM personalization.** Server Action sends `{org_name, ntee_code, state, total_revenue, mission_text, programs[]}` (where mission_text and programs may be empty for low-context orgs) to Claude Sonnet 4.6 (`claude-sonnet-4-6`) via Vercel AI Gateway, with a documented prompt (see Prompt Spec below). Fallback model: Claude Haiku 4.5 (`claude-haiku-4-5-20251001`) for cost or quota. Output: 3-paragraph email draft + suggested subject line, validated with **Zod schema** via `generateObject` (not streaming — deterministic store-and-edit workflow). One automatic retry if Claude returns malformed JSON; structured error shown to user after two failures. UI shows a skeleton 'Generating...' screen while waiting. Editable inline before send.

**Send/track (Gmail).** OAuth via Google Cloud project in Testing mode, scopes `gmail.send` + `gmail.readonly` + `gmail.modify`, Jason as sole test user, 7-day re-consent. Send button on each draft routes through Jason's real inbox via `users.messages.send`, with `In-Reply-To` and `Message-ID` headers preserved for thread tracking. Reply tracking: on page load of the Sent view, poll Gmail History API for the `lead-gen` label (applied to every outbound send) using the persisted `historyId` cursor. Page-load polling — no Vercel Cron required, stays on Hobby tier. Gmail label strategy prevents false positives from unrelated inbox activity. If `historyId` gap exceeds 7 days, trigger full re-sync via `messages.list`.

**LinkedIn manual capture (week 3).** Minimal Chrome extension MV3. Manifest: `host_permissions: ["https://www.linkedin.com/in/*", "<APP_URL>"]`, `permissions: ["activeTab", "storage"]`. Content script + popup. Click extension button → reads visible name, title, current company, profile URL from the active tab → POSTs to `<APP_URL>/api/contacts` with a long-lived bearer token stored in `chrome.storage.local`. The bearer token is generated once in the app's settings UI, copy-pasted into the extension popup. No automation, no DOM crawling, no auto-navigation. The extension only does anything when Jason explicitly clicks it on a profile page he is already viewing.

**Storage (Postgres data model).**

| Table | Columns |
|---|---|
| `orgs` | `id`, `ein`, `name`, `ntee_code`, `state`, `total_revenue`, `propublica_url`, `mission_text` (nullable), `programs_json` (nullable), `created_at`, `enriched_at` |
| `contacts` | `id`, `org_id` (FK), `name`, `title`, `linkedin_url`, `email` (nullable), `created_at` |
| `drafts` | `id`, `org_id` (FK), `contact_id` (FK, nullable), `subject`, `body`, `model`, `prompt_version`, `created_at` |
| `sends` | `id`, `draft_id` (FK), `gmail_message_id`, `gmail_thread_id`, `sent_at` |
| `replies` | `id`, `send_id` (FK), `gmail_message_id` (unique), `received_at`, `snippet` |
| `api_tokens` | `id`, `name`, `token_hash`, `created_at`, `last_used_at` |
| `settings` | `id` (always 1), `history_id`, `gmail_token_refreshed_at`, `gmail_token_status` |

UNIQUE constraints: `orgs(ein)`, `sends(gmail_message_id)`, `contacts(linkedin_url)`, `replies(gmail_message_id)`. Indexes: `contacts(org_id)`, `drafts(org_id)`.

### Prompt Spec (week 1 v1)

```
System: You are helping Jason draft a personalized cold email to a non-profit
about his volunteer match app, [VolApp]. Tone: warm, specific, low-pressure,
no marketing hype. Length: 3 short paragraphs. Open by naming something
specific about the org (use mission_text if available, otherwise NTEE category
+ state). Middle paragraph: one concrete way [VolApp] could help with a
problem orgs of this shape commonly have. Close with: a soft ask for a
15-minute conversation. Sign off as Jason. Do NOT use "I hope this email
finds you well," any superlatives, or any em dashes.

User: <org JSON payload>

Output JSON: { "subject": "...", "body": "..." }
```

Prompt versioned in `drafts.prompt_version` so prompt iterations are tracked against reply rates.

## Known Sharp Edges

These are not blockers but will bite if you don't expect them:

- **IRS 990 XML lookup by EIN.** The IRS S3 bucket (`s3://irs-form-990`) is organized by year + filing index, not by EIN. Per-EIN access requires either downloading the year's index file (small, JSON) and looking up the filing manifest, OR using ProPublica's per-org XML download endpoint when available. Plan: try ProPublica's per-org endpoint first; fall back to year-index lookup.
- **Gmail History API cursor management.** Polling uses `historyId` as cursor. Gaps > 7 days require full re-sync via `messages.list`. Persist last-known `historyId` and handle the gap-recovery branch from day one.
- **Refresh token silent expiry.** In Gmail OAuth Testing mode, refresh tokens expire after 7 days. The page-load poller will break silently when this happens. Implementation: check `settings.gmail_token_refreshed_at` on every Sent view load; if age > 6 days, show a 'Reconnect Gmail' banner (do not silently fail). Store token refresh timestamp in `settings` table on every successful OAuth refresh.
- **Vercel Cron tier.** ~~Resolved: switched to page-load polling (Gmail History API called when Sent view loads). No cron required; stays on Hobby tier.~~
- **NTEE code accuracy.** Verify codes against IRS reference before Day 1 (Open Question #2). Animal welfare = D20. Volunteer-services orgs cluster in T-series (philanthropy/voluntarism) and P-series (human services); pick the right ones for your target.

## UI Design Decisions

Design decisions made during /plan-design-review on 2026-04-26. These are binding for implementation.

### View Architecture (Revised from 3 views → 2 views)

**Decision: No separate Drafts view. Reduce to 2 primary views: Search and Sent.**

Original plan said "Search, Drafts, Sent/Tracking." The Drafts view is removed. Instead:

- **Search view:** Left filter panel + results table. Clicking any org row opens the Draft panel (slide-over).
- **Draft panel:** Right-side shadcn `<Sheet>` (slide-over drawer). Org context on the left half, editable draft on the right half. Opens over the Search results — search state preserved behind it. Closed by clicking outside or pressing Esc.
- **Sent view:** Full-width tracking table. Replied rows sorted to top.

Sidebar nav: Search | Sent | Settings (small, at bottom).

This forces intentionality: Jason selects one org, reviews its context, edits the draft, sends, closes the panel, picks the next org. The tool is designed for "structured first 20 outreaches," not batch sends.

### Search → Draft Flow

**Decision: Click org row → open shadcn `<Sheet>` slide-over panel from the right.**

The Sheet panel is ~60% of viewport width. It sits over the search results (which remain visible and interactive behind the panel). Closing the Sheet returns to search with filters intact. No new page route. No modal.

Sheet contents:
- Header: org name (18px bold) + NTEE badge + "Close" button (X)
- Left half of sheet body: org context (state, revenue, mission text, 990-sourced named contact, "To:" field)
- Right half: subject line input + email body textarea (editable inline)
- Footer: "Saved" indicator (left) + "Generating..." or model badge (center) + "Regenerate" + "Send via Gmail" buttons (right)

### Draft Auto-Save

**Decision: Auto-save draft body to Postgres on 1-second debounce.**

When Jason edits the subject line or body, a debounced server action (`updateDraft`) writes the changes to `drafts.body` and `drafts.subject`. Footer shows "Saved ✓" for 2s after each save, then clears. If auto-save fails, footer shows "Save failed — check connection" in red.

This means drafts survive panel close, page refresh, and browser crash. The draft for each org persists until the org is sent.

### Settings UI Spec

The Settings view is accessible from the sidebar nav ("Settings" link at bottom). Single-purpose page:

- **Gmail section:** OAuth status (connected email, token age, "Reconnect" button). Always shown.
- **API token section:** Shows existing token name + created date + last used. "Regenerate token" button (with confirmation dialog). Copy-to-clipboard button. Instructions: "Paste this token into the Chrome extension popup."
- **Week cap section:** Shows current week's send count + 50-email cap. "Reset manually" button (admin escape hatch for testing).

No other settings in v1.

### Responsive & Accessibility

**Responsive:** Desktop-optimized. Target viewport: 1280px+ (MacBook Pro default). Minimum supported: 1024px wide. For screens < 768px: show a centered banner "This tool is designed for desktop. Please use a laptop or desktop browser." No mobile layout work in v1.

**Accessibility:** shadcn/ui component ARIA baseline only. No custom a11y work. shadcn provides correct keyboard nav, focus management, and ARIA roles for all its primitives (Select, Dialog, DropdownMenu, Table). Custom components (reply expansion, 990 badge) do not require explicit ARIA in v1.

**Color contrast:** Use shadcn/ui default color tokens. Do not override with low-contrast custom colors. Geist on white at 14px regular passes WCAG AA contrast automatically.

### Sent View Sort Order

**Decision: Replied emails sort to top, green badge, inline reply expansion.**

Rationale: the tool's purpose is "quality conversations," not email archiving. When Jason opens the Sent view, his job is to identify replies worth continuing. Chronological sort would bury hot leads.

## Eng Review Pass 2 — Resend Pivot + Hardening (2026-04-26)

This pass replaces the Gmail OAuth subsystem with Resend.com (sender domain
`volunteerready.org` already warmed and live), adds CSV export, and lands 13 hardening

### Architectural Pivot: Resend for send + Resend Inbound for reply tracking

- **Send:** Resend API from `jason@volunteerready.org`, `Reply-To: replies+<verp_token>@replies.volunteerready.org`. Use a `replies.` SUBDOMAIN for inbound MX to avoid colliding with existing volunteerready.org mail flow (Codex finding #1).
- **Reply tracking:** Resend Inbound webhook receives reply, app stores it AND auto-forwards a copy to `jasshultz@gmail.com` so Jason reads replies in his normal Gmail. Forwarder rewrites headers to preserve original sender as Reply-To, so Jason's Gmail "Reply" goes to the right person.
- **Zero Gmail OAuth.** All `/api/auth/gmail/*`, `settings.gmail_token_*`, History API polling, label strategy, reconnect banner deleted.
- **VERP token:** `crypto.randomUUID()` stored in `sends.verp_token` (unique index), looked up on inbound webhook.
- **Webhook security:** Signature verification via Resend's `Svix-Signature` header on every webhook; reject unsigned/invalid.
- **Webhook idempotency:** `webhook_events` table, unique index on Resend `event_id`. Replays no-op.
- **Forwarder loop prevention (4 layers):** `X-Auto-Forwarded: volunteerready` header check + per-`send_id` count cap (5 max) + own-domain sender allowlist (block forwards from volunteerready.org or jasshultz@gmail.com) + global circuit breaker (Postgres-backed counter; halt all forwarding if >20 forwards in last 60s).

### Critical Mail-System Hardening (from Codex outside voice)

- **Suppression list.** New `suppressions` table: `(email | domain, reason: bounced|complained|unsubscribed, suppressed_at)`. On `email.bounced` (hard) or `email.complained` webhook, insert. `sendDraft()` checks suppression before calling Resend. Burns warmed domain if skipped — non-negotiable.
- **Outbound idempotency.** Each send attempt gets a UUID `idempotency_key` BEFORE the Resend API call, persisted in `sends.idempotency_key` (unique). Pass to Resend as `Idempotency-Key` header. Retries with same key are no-ops at Resend. Prevents duplicate sends on response timeout after provider acceptance.
- **Conversation thread state.** `sends` table adds `message_id` (Resend response), `in_reply_to`, `references`. `replies` table adds same headers. Enables multi-touch follow-up sequencing later. `contacts` table adds `replied_at` (derived: most recent reply across all sends to this contact) and `do_not_contact` boolean.
- **Reply classifier.** Before marking a reply as "Replied" in Sent view, classify via headers: `Auto-Submitted: auto-replied`, `X-Autoreply`, `Precedence: bulk|junk|list`, DSN content-type `multipart/report`. Auto-classified replies get `replies.classification IN ('ooo','dsn','autoresponder','bulk')` and do NOT trigger the green "Replied" badge. They show as a separate gray "Auto-reply" badge with snippet.
- **PII redaction in logs.** Pino `redact: ['*.body', '*.html', '*.text', '*.snippet', 'req.headers.authorization', 'req.headers.cookie']`. Email content never reaches Axiom in raw form.
- **Send-to-self loopback smoke test in CI.** Vitest integration test sends a real email via Resend test mode (or sandboxed key) to a test inbox, verifies webhook signature path, dedup, suppression, and forwarder logic end-to-end. Catches MX, MIME, and signature regressions that unit tests miss.
- **Extension token origin check.** `/api/contacts` requires both bearer token AND `Origin: chrome-extension://<EXTENSION_ID>` header. Reduces blast radius of leaked token.
